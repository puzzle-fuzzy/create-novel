# 小说生成智能体 — 深层架构缺陷分析

> 视角：智能体架构设计专家
> 日期：2026-05-24
> 审查对象：novel-generator 全部源码 + 「数字」项目实际输出

---

## 零、核心论断

**这个系统不是一个"智能体"，而是一个"带 LLM 调用的批处理脚本"。**

它不具备智能体的三个基本特征：
1. **感知（Perception）**：系统对已生成内容没有任何理解，只做字数统计
2. **推理（Reasoning）**：没有反思、没有规划修正、没有质量判断
3. **行动（Action）**：每次行动都是一锤子买卖，写完即存，不可回溯

它本质上是一个 `for` 循环 + LLM API 调用的线性管道（Pipeline）。
问题不在于"有哪些 bug 可以修"，而在于**架构范式本身需要升级**。

下面从七个维度展开。

---

## 一、它不是智能体——缺乏反馈闭环

### 1.1 现状

当前的数据流是完全单向的：

```
Config → WorldBible → Premise → VolumeOutlines → Chapters → Compile
```

每一步都是"生成 → 保存 → 下一步"。没有一步会回头看。

### 1.2 问题

真正的创作是一个**不断自我修正**的过程。人类作家写完一章后会：
- 重读，发现前后矛盾
- 修改前面的伏笔来配合后面的剧情
- 发现某个角色写得太平，回去加戏
- 发现节奏不对，删减或扩充某些段落

当前系统**完全不具备这种能力**。一旦一个章节写好被保存，它就变成了"事实"，即使后续章节已经与之矛盾。

### 1.3 实际证据

- v01_c002 和 v01_c003 开头完全相同（"雨水如注，像无数条鞭子"）— 如果系统有自检，绝不会允许这种事发生
- outline 规划了 8 卷，实际只生成了 6 卷就标记 completed — 没有校验，系统甚至不知道自己少做了 2 卷
- v06 结尾写出了"系统重启中……格式化旧世界……"这种元叙事，与全书仙侠设定产生风格断裂 — 没有人审查

### 1.4 解决方案：引入 Agent Loop

改造核心架构为 **"生成 → 评估 → 修正" 循环**：

```
┌─────────────────────────────────────────────┐
│              Agent Controller                │
│                                             │
│   ┌─────┐    ┌──────┐    ┌───────┐         │
│   │ Plan │───→│ Write │───→│Review │──┐      │
│   └─────┘    └──────┘    └───────┘  │      │
│       ↑                              │      │
│       └──────────────────────────────┘      │
│              (质量不达标则重写)                │
│                                             │
│   ┌──────────┐                              │
│   │ Consistency │ (持续运行)                  │
│   │ Checker    │                              │
│   └──────────┘                              │
└─────────────────────────────────────────────┘
```

- **Review Agent**：每章写完后，用 LLM 对齐进行质量打分（情节连贯性 / 角色一致性 / 文笔质量 / 开头结尾），低于阈值自动重写
- **Consistency Checker**：维护一个"事实数据库"（人物位置、已揭示信息、关系变化），每章写完后更新，新章生成前查询

## 二、它没有记忆——上下文管理的根本缺陷

### 2.1 现状

writer.ts 中传递给 LLM 的上下文只有：
- 世界设定书的前 3000 字（硬截断）
- 本卷的 summary
- 本章的大纲
- **前一章**的"摘要"（实际是 head 300字 + tail 200字的粗暴拼接）
- **前一章**的尾部 800 字

### 2.2 问题

一个 20 万字的长篇有 60 章。写到第 30 章时：
- 第 1-28 章的内容对 LLM 来说**完全不存在**
- 第 29 章的"摘要"只是头尾拼接，丢失了中间所有关键情节
- 世界设定书被截断到 3000 字，大纲可能有 1 万字

这就像让一个作家写第 30 章，但只给他看第 29 章的前几段和最后几段，以及一个被撕掉 70% 的设定手册。

### 2.3 实际证据

- "归零"在全书出现 167 次，但"变量"只出现 22 次 — 说明后面章节基本忘了"变量"这个核心设定，只记得"归零"
- 云长老只出现 75 次（叶尘的 1/26），对于大纲中定义的"主要配角"来说戏份严重不足 — 后面的章节忘了这个角色的存在
- "系统""数据""代码"等数字世界相关关键词在 v06 才突然密集出现，前面几乎不用 — 世界观的核心设定（"数字牢笼"）没有贯穿全书

### 2.4 解决方案：分层记忆架构

```
┌─────────────────────────────────────┐
│          Memory System               │
│                                     │
│  Layer 1: 长期记忆（Permanent）      │
│  ├── 世界设定书（完整）               │
│  ├── 角色档案（动态更新）             │
│  └── 全书时间线（事件流）             │
│                                     │
│  Layer 2: 中期记忆（Rolling Window） │
│  ├── 最近 5 章的结构化摘要            │
│  ├── 活跃伏笔列表                    │
│  └── 当前场景角色状态                 │
│                                     │
│  Layer 3: 短期记忆（Immediate）      │
│  ├── 前一章完整尾部（2000字）         │
│  └── 当前章的大纲 + 写作指令          │
│                                     │
│  动态组装策略：                       │
│  根据当前章大纲，从各层检索最相关的    │
│  上下文，组装成不超过模型窗口的 prompt  │
└─────────────────────────────────────┘
```

关键实现：
- **每章写完后用 LLM 生成结构化摘要**（而非头尾截取），包含：关键事件 / 角色状态变化 / 新信息揭示 / 伏笔埋设
- **维护 `state.json`**：记录每个角色当前在哪、什么状态、知道什么信息、与谁的关系如何
- **伏笔追踪器**：大纲中标注的伏笔在写完后标记为"已埋"或"已收"，未收的伏笔在后续章节的 prompt 中提醒

## 三、它把"创作"当"生成"——Prompt 工程的系统性缺陷

### 3.1 现状

当前 prompt 模板是静态的字符串拼接。`buildChapterSystemPrompt()` 生成一个固定的系统提示，`buildChapterUserPrompt()` 拼接固定结构的信息。

### 3.2 问题

#### (a) Prompt 与写作阶段无关

第 1 章（开篇）和第 50 章（高潮）用的是同一套 prompt 模板。但：
- 开篇需要：建立世界观、引出悬念、塑造角色第一印象
- 高潮需要：回收伏笔、多线汇聚、情绪爆发、节奏加速
- 过渡需要：节奏放缓、角色互动、信息铺垫

当前 prompt 没有任何"阶段感知"能力。

#### (b) 没有 few-shot 示例

prompt 里只有指令，没有给出"好的开头长什么样"、"好的对话长什么样"。对于 flash 级别的模型，没有示例就等于让它在黑暗中摸索。

#### (c) 写作指令过于笼统

"充分展开每个关键场景" — 什么叫"充分"？LLM 的理解可能完全不同于用户期望。

### 3.3 解决方案：动态 Prompt 构建

```typescript
// 伪代码
function buildDynamicPrompt(chapter, context) {
  const phase = identifyPhase(chapter); // 'opening' | 'rising' | 'climax' | 'transition' | 'resolution'
  
  // 1. 根据阶段选择不同的写作策略
  const strategy = PHASE_STRATEGIES[phase];
  
  // 2. 注入与本章相关的 few-shot 示例
  const examples = selectExamples(chapter.mood, phase);
  
  // 3. 动态注入"禁忌清单"（基于历史分析）
  const taboos = buildTabooList(context.previousChapters);
  // 例如：检测到最近3章都用环境描写开头，就加入"不要以环境描写开头"
  
  // 4. 注入未回收的伏笔提醒
  const pendingForeshadows = context.getUnresolvedForeshadows();
  
  return assemblePrompt(strategy, examples, taboos, pendingForeshadows);
}
```

## 四、它没有世界观一致性保障——设定漂移问题

### 4.1 现状

世界设定书生成一次后就被固化了。写作过程中，LLM 会不断"发明"新设定（新地名、新角色、新能力），这些新设定可能与原有设定矛盾。

### 4.2 问题

当前系统对 LLM 输出的内容**零约束**。LLM 可以：
- 让一个应该在中州的角色突然出现在东荒
- 给主角安排一个设定书中不存在的招式
- 改变力量体系的规则
- 编造一个与历史大事记矛盾的事件

而且这些"漂移"一旦发生，后续章节会基于错误信息继续"漂移"，形成**级联错误**。

### 4.3 实际证据

- outline 的 premise 中明确写"天地灵气被视为流动的数据流"，但前 3 卷几乎不提这个设定，到 v06 才突然出现"系统重启""格式化"等数字世界概念 — 严重的设定漂移
- 云长老在设定中是"亦敌亦友的天才"，但在实际正文中更像一个普通的宗门长老，"天才"和"亦敌亦友"特质不明显

### 4.4 解决方案：事实约束层

```
┌────────────────────────────────────┐
│       Fact Constraint Layer         │
│                                    │
│  输入：LLM 生成的章节内容            │
│                                    │
│  Step 1: 实体提取                   │
│  从生成的章节中提取所有：             │
│  - 人物名、地名、组织名              │
│  - 能力/招式名                      │
│  - 时间/地点描述                    │
│                                    │
│  Step 2: 一致性校验                 │
│  对比世界设定书 + state.json：       │
│  - 角色是否应该在当前位置？           │
│  - 能力是否符合力量体系？             │
│  - 事件是否与时间线矛盾？             │
│                                    │
│  Step 3: 新实体注册                 │
│  LLM 新创造的合理实体加入 state.json │
│  不合理的实体标记为矛盾               │
│                                    │
│  输出：一致性报告 + 是否需要重写      │
└────────────────────────────────────┘
```

实现方式：不一定要用 LLM 做实体提取。对于仙侠小说这种强类型化的作品，可以用正则 + 关键词匹配做初步过滤，再用 LLM 做语义层面的校验。

## 五、它的规划是脆弱的——大纲生成的问题

### 5.1 现状

大纲生成流程：
1. 生成世界设定书（一次性，不再更新）
2. 生成故事总体概要（一次性）
3. 逐卷生成大纲（每卷独立，只拿前一卷 summary）

### 5.2 问题

#### (a) 世界设定书不会进化

真实创作中，作者会随着写作深化对世界的理解。当前系统的世界设定书是"冻结"的——写完第一版就永不更新。

#### (b) 卷级大纲之间信息损失严重

每卷只拿到前一卷的 summary（200-300字）。对于 10 章 × 3500字 = 35000字 的内容来说，300字的 summary 信息压缩比高达 99%。

#### (c) 章级大纲缺少跨章约束

10 章的大纲是一次性生成的，LLM 可能会在第 3 章安排一个需要第 8 章回收的伏笔，但第 8 章的 summary 中完全没有提及。

#### (d) 大纲与实际写作之间没有反馈

大纲生成后就是"圣旨"，即使写作过程中发现大纲的某个设定不合理，也不能修改。

### 5.3 实际证据

- outline.json 规划 8 卷，实际只生成 6 卷 — planner 在第 6 卷后静默失败
- v06 结尾出现"系统重启中""格式化旧世界"等元叙事内容，这与大纲规划的"第八卷：重启"完全不符 — 说明 v06 写作时丢失了"后面还有两卷"的信息，提前写了结局
- 前三卷几乎不碰"数字牢笼"这个核心设定，说明卷级大纲生成时没有足够的全局约束来保证主题贯穿

### 5.4 解决方案：动态规划 + 大纲修正

```
┌─────────────────────────────────────┐
│      Adaptive Planning System        │
│                                     │
│  Level 1: 全书蓝图（Premise）        │
│  → 只定义方向，不定义细节             │
│  → 每写完一卷后回顾，允许微调方向      │
│                                     │
│  Level 2: 卷级规划（Volume Plan）    │
│  → 写一卷前才生成该卷的详细大纲       │
│  → 基于已写内容的实际状态             │
│  → 而非基于一开始的预测               │
│                                     │
│  Level 3: 章级规划（Chapter Plan）   │
│  → 写每章前动态调整                   │
│  → 基于前一章的实际输出（而非计划输出）│
│                                     │
│  反馈机制：                          │
│  每写完一卷 → Review Agent 评估      │
│  → 发现大纲偏差 → 修正后续卷的大纲    │
│  → 发现伏笔遗漏 → 插入后续章节计划    │
└─────────────────────────────────────┘
```

核心思想：**不要一次性规划 60 章的细节，而是"走一步看一步"**。只规划近期（1-2卷）的详细大纲，远期只保持方向。

## 六、它的质量度量是盲目的——没有评估体系

### 6.1 现状

当前系统衡量"成功"的唯一指标是：
- `completedChapters` / `totalChapters` — 完成了多少章
- `totalWords` — 总字数
- `totalTokensUsed` — 花了多少钱

### 6.2 问题

这些指标**全部与质量无关**。系统完全不知道：
- 生成的内容好不好看
- 前后是否一致
- 角色是否立体
- 剧情是否有推进
- 读者会不会想继续读

一个智能体系统如果不能度量自己的输出质量，就不可能自我改进。

### 6.3 解决方案：多维质量评估框架

```
┌────────────────────────────────────────┐
│     Quality Metrics Framework           │
│                                        │
│  维度 1: 客观指标（代码可计算）          │
│  ├── 字数是否达标                       │
│  ├── 开头首字是否与前章重复              │
│  ├── 对话占比是否在设定范围              │
│  ├── 结尾套路化程度（与黑名单的相似度）   │
│  └── 章节间词汇重叠度（过高=重复）       │
│                                        │
│  维度 2: LLM 评估（需额外 API 调用）     │
│  ├── 情节连贯性 (1-10)                  │
│  ├── 角色一致性 (1-10)                  │
│  ├── 场景描写质量 (1-10)                │
│  ├── 对话自然度 (1-10)                  │
│  ├── 大纲完成度 (1-10)                  │
│  └── 章节吸引力 (1-10)                  │
│                                        │
│  维度 3: 跨章节一致性                    │
│  ├── 设定矛盾检测                       │
│  ├── 角色位置连续性                     │
│  ├── 伏笔埋设与回收追踪                 │
│  └── 时间线一致性                       │
│                                        │
│  用途：                                 │
│  - 低于阈值的章节自动重写               │
│  - 生成质量报告供人类审查               │
│  - 积累数据用于后续优化 prompt          │
└────────────────────────────────────────┘
```

成本控制：不需要每章都做全维度评估。可以只对客观指标做代码检测（零成本），LLM 评估只抽检或每隔 N 章做一次。

## 七、它没有"人"的角色——缺少 Human-in-the-Loop

### 7.1 现状

当前系统的交互模式是：
1. `init` — 用户回答 18 个问题（但几乎所有问题都有默认值，可以直接回车跳过）
2. `plan` — 全自动，用户看不到大纲，不能修改
3. `write` — 全自动，60 章一次性写完
4. `compile` — 全自动

用户唯一能干预的时机是**写之前**（改 config.yaml），一旦开写就完全失控。

### 7.2 问题

小说创作本质上是**人与 AI 协作**的过程。用户应该能够：
- 在大纲生成后审查、修改、补充
- 在关键章节（开篇、高潮、结局）审查内容质量
- 在发现问题时暂停、回退、修改大纲后继续
- 标记"这章不满意，请重写"
- 插入自己的创作（"这章中间我想加一段主角回忆"）

### 7.3 解决方案：交互式工作流

```
┌──────────────────────────────────────────┐
│     Human-in-the-Loop Workflow            │
│                                          │
│  ┌──────────┐                             │
│  │  Plan    │──→ [人工审查大纲] ──→ ✅/❌  │
│  └──────────┘      用户可以：             │
│                    - 修改章节大纲          │
│                    - 删除/插入章节         │
│                    - 调整角色设定          │
│                                          │
│  ┌──────────┐                             │
│  │  Write   │──→ 每N章暂停 ──→ [人工审查] │
│  └──────────┘      用户可以：             │
│                    - 标记不满意章节        │
│                    - 添加写作指导          │
│                    - 手动编辑已生成内容    │
│                    - 继续/暂停/回退        │
│                                          │
│  ┌──────────┐                             │
│  │ Review  │──→ 生成质量报告 ──→ [人工决策]│
│  └──────────┘      用户可以：             │
│                    - 选择哪些章节重写      │
│                    - 微调重写方向          │
│                    - 确认最终版本          │
└──────────────────────────────────────────┘
```

技术实现：
- Dashboard 中增加"审查模式"
- 每个章节旁增加"重写""编辑""标记问题"按钮
- 大纲页面支持拖拽重排、内联编辑

## 八、它缺少抽象——角色/世界观只是字符串

### 8.1 现状

角色信息存储为：
```typescript
interface Character {
  name: string;
  role: string;
  description: string;  // 自由文本
  personality: string;  // "待定"
  background: string;   // "待定"
  arc: string;          // 自由文本
}
```

世界设定也是一堆自由文本字段。没有结构化、没有约束、没有关系图。

### 8.2 问题

- 角色之间的关系（师徒、情敌、父子）只是 description 中的文字描述，代码无法理解和操作
- 力量体系只是一个逗号分隔的字符串，代码不知道"金丹"比"筑基"高
- 地理关系（东荒在中州的东边）只是文字，代码无法推理"从东荒到中州需要多远"

这意味着所有基于角色关系、力量等级、地理距离的推理都只能靠 LLM 在运行时理解——而 LLM 会遗忘、会搞混。

### 8.3 解决方案：知识图谱化

```typescript
interface CharacterV2 {
  id: string;
  name: string;
  aliases: string[];           // 别名、绰号
  role: 'protagonist' | 'antagonist' | 'supporting' | 'minor';
  
  // 结构化属性
  cultivationLevel: number;     // 当前修为等级（在力量体系中的位置）
  affiliation: string;          // 所属势力
  currentLocation: string;      // 当前位置（动态更新）
  
  // 关系网络
  relationships: Relationship[];
  
  // 状态追踪
  knownInformation: string[];   // 这个角色目前知道什么
  activeGoals: string[];        // 这个角色当前想做什么
  emotionalState: string;       // 当前情绪状态
  
  // 行为模板
  speechPattern: string;        // 说话方式（用于对话生成）
  decisionMakingStyle: string;  // 决策风格
}

interface Relationship {
  with: string;          // 对方角色 ID
  type: string;          // '师徒' | '情敌' | '父子' | '盟友' | ...
  strength: number;      // -10 到 10
  development: string[]; // 关系发展事件列表
}
```

这样做的好处：
- 写每章前可以自动查询"这章出场的角色之间是什么关系"
- 对话生成时可以参考 speechPattern
- 一致性检查时可以校验角色位置是否合理

## 九、改进路线图

### Phase 1：最小可行改进（1-2 天）

不改架构，只加最关键的几件事：

1. **开头重复检测**（代码级，零成本）
   ```typescript
   function checkOpeningDuplication(newContent: string, prevContents: string[]): boolean {
     const first3 = newContent.slice(0, 3);
     return prevContents.slice(-5).some(prev => prev.slice(0, 3) === first3);
   }
   ```

2. **真正的章节摘要**（每章写完后调 LLM 生成，存入 summaries.json）

3. **大纲卷数校验**（生成完后检查实际卷数是否等于配置卷数）

### Phase 2：Agent Loop（3-5 天）

改造核心循环为"生成→评估→修正"：

4. 引入 **Review Agent**：写完一章后让 LLM 打分
5. 引入**质量阈值**：低于阈值自动重写（最多 3 次）
6. 引入 **state.json**：跟踪角色状态、位置、已知信息

### Phase 3：分层记忆（5-7 天）

7. 实现 **三层记忆架构**（长期/中期/短期）
8. 实现**动态 prompt 组装**（根据当前章节需要从各层检索信息）
9. 实现**伏笔追踪器**

### Phase 4：结构化知识（7-14 天）

10. 角色/势力/地理的**结构化建模**
11. **一致性校验器**（基于知识图谱检查矛盾）
12. **动态大纲调整**（每卷写完后基于实际状态调整后续规划）

### Phase 5：Human-in-the-Loop（14+ 天）

13. Dashboard 增加**审查模式**
14. 支持**人工编辑后重写**
15. 支持**章节级回退和重生成**

## 十、总结

| 维度 | 当前状态 | 应有状态 |
|------|---------|---------|
| 架构范式 | 线性 Pipeline | Agent Loop（感知→推理→行动→反馈） |
| 记忆系统 | 无（只看前一章头尾） | 三层记忆（长期/中期/短期） |
| 质量控制 | 无 | 多维评估 + 自动重写 |
| 规划方式 | 一次规划60章 | 滚动规划（只详细规划近1-2卷） |
| 一致性保障 | 无 | 事实约束层 + 知识图谱 |
| 人类参与 | 仅初始化时 | 全程可介入 |
| 知识表示 | 自由文本 | 结构化 + 关系图 |
| 评估度量 | 字数/章数 | 多维质量指标 |

**一句话总结**：当前系统能"生成"一部小说，但不能"写好"一部小说。从"能跑"到"好用"的差距，本质上是**从 Pipeline 到 Agent 的范式升级**。
