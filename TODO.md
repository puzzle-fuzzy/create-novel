# 小说生成智能体 — 缺陷与改进清单

> 生成时间：2026-05-24
> 基于对已完成项目「数字」（6卷60章，约21.8万字）的全面审查

---

## 一、架构设计层面

### 1.1 缺少「上下文窗口管理」机制 🔴 严重

**现状**：writer 只传递前一章的摘要（粗暴截取前300字+后200字拼接）和尾部800字；世界设定书只取前3000字。

**问题**：
- 写到第30章时，第1-28章的细节（人名、地点、伏笔）全部丢失
- 摘要不是真正的"摘要"，只是头尾拼接，中间大量关键情节被丢弃
- 世界设定书可能有1万+字，但只给了前3000字，后面的势力、地理、角色细节全部缺失

**实际影响**：v01 的 c002 和 c003 开头几乎一样（"雨水如注，像无数条鞭子"），说明 LLM 根本没有足够的上文来理解"上一章已经用雨开头了"

**方案**：
- 写完每章后调 LLM 生成200字结构化摘要（记录关键情节、角色状态、伏笔）
- 维护 `summaries.json`，写作时传入最近5-10章的摘要
- 世界设定书按章节需要的相关段落动态传入，而非硬截3000字

---

### 1.2 没有「大纲连贯性校验」🟡 中等

**现状**：大纲逐卷独立生成，每卷只拿上一卷的 summary，章级大纲之间没有交叉校验。

**问题**：
- 第一卷10章的 summary 可能有重复、矛盾、时间线混乱
- 角色在不同卷之间的行为可能不一致
- 伏笔在前面埋了，后面忘记收

**方案**：
- 全部大纲生成完后，增加一步"大纲审查"（批量检查重复、矛盾、遗漏）
- 用更高级的模型进行审查
- 生成审查报告，标注问题章节并自动修正

---

### 1.3 没有「质量审查」环节（Critique/Review）🔴 严重

**现状**：流水线是 `Plan → Write → Save`，完全是单向推进，没有回看机制。

**问题**：
- 章节写完后没有任何质量检查（字数是否达标？开头是否重复？剧情是否偏移大纲？）
- 没有自动修章机制——如果一章质量差，它就被原样保留

**方案**：
- 增加 Critique 步骤：写完一章后让 LLM 打分（0-10），低于阈值自动重写
- 代码层面检测：开头首字是否与前一章重复、字数是否达标
- 检测不通过则自动重试（最多3次）

---

## 二、内容质量层面

### 2.1 章节开头重复 🔴 严重

**现状**：v01 的 c002-c010 共9章全部以"雨"开头，其中 c002 和 c003 首句完全相同。

```
c002: 雨水如注，像无数条鞭子抽打着东荒的苍穹。
c003: 雨水如注，像无数条鞭子抽打着东荒的苍穹。
```

**根因**：LLM（glm-4-flash）是轻量模型，对 prompt 中"不要重复"的指令遵守度不高。仅靠 prompt 文字约束不够。

**方案**：
- 代码层面：写完后检查前3字是否与前一章相同，如果是则自动重试（最多3次）
- prompt 中已增加约束（已完成），但需要代码层面的兜底

---

### 2.2 章节结尾套路化严重 🟡 中等

**现状**：v01 的10章结尾几乎全是"宏大叙事+悬念句"模式：

- "一切，才刚刚开始。"
- "太虚界，终究不会平静。"
- "他是零。他是终结。也是开始。"
- "东荒的夜，才刚刚开始。"

**问题**：读者连续看60章会严重审美疲劳。

**方案**：
- prompt 中给出"避免的结尾模式"黑名单（如"一切才刚刚开始"等套路句式）
- 给出多样化的结尾范例（对话式收尾、悬念对话、动作定格、情绪留白等）
- 代码层面检测结尾相似度，过高则重试

---

### 2.3 章节字数偏短 🟢 低

**现状**：各卷平均 3400-4000 字/章，网文通常 5000-8000 字/章。

**方案**：
- config 中已将 targetWords 从 2500 调至 4000（已完成）
- 长期可考虑按章节重要性差异化设置（高潮章5000-6000，过渡章3000-4000）

---

### 2.4 大纲 targetWords 千篇一律 🟢 低

**现状**：大纲 JSON 里每章的 `targetWords` 全是 2500，没有根据章节内容重要性做差异化。

**方案**：
- 大纲生成 prompt 中要求 LLM 按章节重要性设置不同 targetWords
- 高潮章节更长，过渡章节可以短一些

---

## 三、生成鲁棒性层面

### 3.1 错误恢复粒度太粗 🟡 中等

**现状**：`progress.json` 只记录 `completedChapters` 和 `lastWrittenChapter`。如果写作在某一章中途失败（API 超时），这章会完全丢失。

**方案**：
- 写章前先写临时文件（`.tmp`），写完并验证通过后才重命名为正式文件
- 记录每章字数到 progress，方便断点诊断

---

### 3.2 JSON 解析修复过于激进 🟢 低

**现状**：`repairJSON` 里的正则 `text.replace(/([{,]\s*)(\w+)\s*:/g, '$1"$2":')` 会把所有未加引号的 key 加上引号，但如果 JSON value 里面自然出现 `word:` 这种模式，也会被错误修改。

**方案**：
- 改用更保守的 JSON 修复策略
- 或者直接要求 LLM 输出更严格的 JSON（使用 structured output / function calling）

---

### 3.3 模型选择全部用 flash 级别 🔴 严重

**现状**：`.env` 里三个模型全部是 `glm-4.7-flash`：
```
LLM_MODEL=glm-4.7-flash
LLM_PLANNER_MODEL=glm-4.7-flash
LLM_WRITER_MODEL=glm-4.7-flash
```

**问题**：大纲规划和正文写作用同一个轻量模型，对于 20 万字长篇来说质量堪忧。v01 的重复开篇问题很大程度上就是因为 flash 模型的创作多样性不足。

**方案**：
- Planner（大纲规划）用更强的模型（如 glm-4-plus），规划质量决定全书质量
- Writer（正文写作）可以继续用 flash 控制成本，但建议提升 temperature 到 0.9+
- 建议在 README 中明确推荐模型分级策略

---

### 3.4 限流控制简陋 🟡 中等

**现状**：每章写完硬编码 `setTimeout(resolve, 2000)`。

**问题**：如果 API 限流窗口变化，或者换了一个更严格的 API，2秒可能不够。

**方案**：
- 实现指数退避（exponential backoff）
- 检测 429 状态码后按 `Retry-After` header 等待
- 允许用户配置请求间隔

---

### 3.5 只生成了6卷但大纲规划了8卷 🟡 中等

**现状**：config 设定8卷80章，但实际只生成了6卷60章就标记 completed。

**问题**：planner 在第6卷之后的生成可能失败了但被静默跳过了，或者 volumeCount 在某处被覆盖。

**方案**：
- planner 生成大纲后校验实际卷数是否与配置一致
- 不一致时给出明确警告并尝试补全

---

## 四、工程层面

### 4.1 API Key 明文存在 `.env` 中且被 git 追踪 🔴 严重

**现状**：`.gitignore` 没有排除 `.env`，`.git` 目录存在。

**方案**：
- `.gitignore` 加入 `.env`
- 如果已经 commit 过，需要 `git filter-branch` 或 `git rm --cached .env` 清除历史
- 考虑使用环境变量或 secret manager

---

### 4.2 `loadEnvFile` 手工解析 🟢 低

**现状**：手动解析 `.env` 文件，不支持多行值、转义引号等边界情况。

**方案**：Bun 原生支持 `.env`，直接删除 `loadEnvFile` 函数，改用 Bun 的自动加载。

---

### 4.3 字数统计用 `content.length` 不准确 🟢 低

**现状**：中文里 `content.length` 统计的是字符数不是字数（标点符号、空格、换行都算进去了）。

**方案**：
- 用正则 `content.match(/[\u4e00-\u9fff]/g)?.length || 0` 统计纯汉字数
- 或用 `content.replace(/[^\u4e00-\u9fff]/g, '').length`

---

### 4.4 Dashboard 没有鉴权 🟢 低

**现状**：API 直接暴露，没有任何认证机制。

**方案**：如果只是本地使用可以暂不处理；如果要部署则需增加认证。

---

## 五、优先级排序

### 🔴 高优先级（直接影响小说质量）

1. **[1.1] 建立章节摘要系统** — 解决长篇上下文丢失的根本问题
2. **[2.1] 开头重复检测 + 自动重试** — 代码层面强制保证开篇多样性
3. **[3.3] 大纲和写作用不同等级的模型** — 规划质量决定全书质量
4. **[1.3] 增加质量审查环节** — Critique 打分 + 不达标自动重写
5. **[4.1] API Key 安全** — `.gitignore` 加入 `.env`，清除 git 历史

### 🟡 中优先级（提升整体体验）

6. **[1.2] 大纲连贯性校验** — 全书大纲生成后的交叉审查
7. **[2.2] 结尾套路化** — prompt 增加结尾黑名单和多样化范例
8. **[3.1] 错误恢复粒度** — 临时文件写入机制
9. **[3.4] 限流控制** — 指数退避 + Retry-After
10. **[3.5] 卷数校验** — 确保生成卷数与配置一致

### 🟢 低优先级（锦上添花）

11. **[2.3] 字数差异化** — 按章节重要性设置不同 targetWords
12. **[2.4] 大纲 targetWords 差异化**
13. **[3.2] JSON 解析修复策略优化**
14. **[4.2] 使用 Bun 原生 .env 加载**
15. **[4.3] 准确的字数统计**
16. **[4.4] Dashboard 鉴权**
